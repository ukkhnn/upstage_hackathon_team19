## D-3 (5/28 목)

> 아래 오전/오후/저녁 구분은 임의 분할이며 미정. 작업 순서 자체는 유지하되 실제 시간 배치는 두 사람 일정에 맞게 조정.

### (시간대 미정) 단독 테스트 단계

6. n8n 사전 학습 강의 이수 (둘 다 같이)
7. Upstage API Document Parse 단독 테스트 — 아무 PDF 한 개 넣고 텍스트로 변환되는지만 확인. 여기서 결과 보고 정책 PDF 포맷을 결정해야 함
8. Upstage Solar LLM API 단독 테스트 — Postman이나 n8n HTTP 노드로 간단한 호출 한 번
9. Upstage Information Extract 단독 테스트 — 짧은 텍스트 하나로 추출 스키마 동작 확인

### (시간대 미정) 설계 합의 단계 — 가장 중요한 의사결정 구간

10. 데모 시나리오 컨셉 합의 — 메인 시연용 1개 + 보조 시연 2~3개. 어떤 케이스를 위험으로 잡을지 미리 정해야 정책 PDF 내용이 결정됨
11. 룰셋 스키마 설계 — 어떤 필드를 뽑아낼지(카테고리·기간·금액·조건). Google Sheets 시트 구조도 같이
12. Information Extract 추출 스키마 작성 — JSON 형태로

### (시간대 미정) 본격 산출물 1차 착수

13. 가상 정책 PDF 1개 초안 작성 (가장 단순한 환불 정책부터) — 룰셋 스키마와 일치하는 문장 구성
14. 데모 시나리오 텍스트 3~5개 초안 작성 — 안전 1개, 주의 1개, 위험 2~3개
15. 비즈니스 기획서 템플릿 다운로드 (해커톤 측 제공 양식 확인)

## D-2 (5/29 금)

> 아래 시간대 구분은 임의 분할이며 미정.

### (시간대 미정) 사전 등록 워크플로우 완성

16. n8n 사전 등록 워크플로우 구현 — Document Parse → Information Extract → Google Sheets 저장까지
17. 사전 등록 워크플로우 end-to-end 테스트 — 13에서 만든 정책 PDF를 실제로 넣어서 룰셋이 시트에 들어가는지 확인
18. 벡터 DB 셋업 + 임베딩 워크플로우 추가 (시간 여유 있을 때. 우선순위 낮음 — 첫 데모는 룰셋 DB만으로도 가능)

### (시간대 미정) 런타임 워크플로우 구현

19. n8n 런타임 워크플로우 구현 1단계 — Webhook 트리거 + 의도 분류
20. n8n 런타임 워크플로우 구현 2단계 — RAG 답변 생성
21. n8n 런타임 워크플로우 구현 3단계 — 답변에서 조건 추출 + 룰셋 대조 + 위험도 산출
22. 런타임 워크플로우 응답 스키마 확정 (지은과 합의해서 Lovable이 받을 JSON 구조 결정)

### (시간대 미정) 발표용 자산 1차 착수

23. 추가 정책 PDF 작성 (교환·배송 — 시간 여유에 따라)
24. 비즈니스 기획서 초안 4페이지 작성 — 문제/타겟/솔루션/BM
25. 발표 자료(피칭 덱) 슬라이드 골격 잡기 — 8~10장 구성 미리

**중요 체크포인트:** 이날 밤까지 end-to-end 한 사이클(고객 문의 입력 → 위험도 산출)이 돌아가야 함. 만약 안 되면 토요일 오프라인 개발일이 디버깅에 다 잡아먹힘.

## 운용 방식

민주와 현욱 둘이 같이 1+3 풀에서 움직이는 거라 작업을 그때그때 분배. 예를 들어 한 사람이 n8n 워크플로우 구현하는 동안 다른 사람은 정책 PDF 다듬고 기획서 쓰는 식. 역할은 미고정이고, 작업 단위로 그날그날 분담.
